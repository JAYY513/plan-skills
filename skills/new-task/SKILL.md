# new-task：任务录入与拆解

## 先读四个文件

ROADMAP.md（找 ▶ 当前里程碑）、TASKS.md、INBOX.md、FINDINGS.md。缺失任一文件 → 提示用户先运行 plan-init。

## 分流判断

用户给出一个事项时，按顺序判断：

1. **是调研结论或重要发现吗？** → 写入 FINDINGS.md（按条目格式：编号、来源、结论含证据、影响标注）。结论影响计划（改变某里程碑可行性或 SPEC 选型）→ 同步在 INBOX.md 登记一条并引用 FINDINGS 编号，告诉用户"已停车，周回顾时裁决"
2. **是失败尝试吗？**（试过某方案行不通）→ 写入 FINDINGS.md，来源标"失败尝试"，写清"试过什么、为什么不行"
3. **它服务于当前里程碑吗？**
   - 是 → 进 TASKS.md（走下面的录入流程）
   - 否 / 不确定 → 写入 INBOX.md，标注 🔴（影响当前里程碑）或 ⚪（不影响），告诉用户"已停车，周回顾时裁决"，**不要**当场改 ROADMAP.md / SPEC.md
4. **是调研吗？** → 进 TASKS.md「调研（限时探针）」区，必须向用户确认时间盒和产出要求（产出默认要求落 FINDINGS.md），缺了就问
5. **用户只是问"下一步做什么"** → 从当前里程碑的验收标准反推，给出 1~2 个候选任务（带 DoD），用户确认后录入

## 录入流程（进 TASKS.md 的任务）

1. **检查粒度**：预估超过 2 天 → 拆成 0.5~2 天的子任务，每个可独立完成
2. **生成 DoD**：每条任务写一条可手动验证的完成标准。
   - 差的 DoD："功能能用"
   - 好的 DoD："点击中断 → 输出立刻停止 → 按钮恢复可发状态，手动验证通过"
3. 追加到 TASKS.md 对应区域（进行中 / 已拆好 / 调研）
4. **检查队列水位**：「进行中」+「已拆好」合计少于 2 个 → 主动从当前里程碑反推，提议补充任务

## 任务完成流程（用户说"做完了"）

1. 逐条核对该任务的 DoD，全部通过才算完成
2. 把任务从「进行中」移到「已完成（待归档）」，标记 ✅ 并写完成日期 + 一句"实际怎么做的"备注
3. **不许删除**——归档是 weekly-review 的职责
4. 过程中踩过的坑、推翻的方案 → 顺手记入 FINDINGS.md
5. 检查队列水位，不足则提议补充

## 纪律

- 不凭空发明任务；拿不准归属就问用户一句，不要猜
- 不修改 ROADMAP.md / SPEC.md——那是周回顾的职责
- 调研结论不落 FINDINGS.md 就不算调研完成
